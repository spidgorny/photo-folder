# Architecture Plan: Automated Photo Upload from Android to S3 (Flutter)

## Overview

This document outlines the architecture for building an **Android application** (using Flutter) that runs once per hour, discovers new photos on the device, and uploads them to the existing S3 bucket used by the **photo-folder** Next.js web UI.

The goal is to replace/supplement manual tools like *S3 Drive* with an automated, background-capable mobile client while evolving the existing web UI into a proper backend API with strong authentication.

## 1. High-Level Architecture

### Components
- **Flutter Android App** (new): Scans local photo library, detects new photos, authenticates, uploads.
- **Next.js Web UI + API** (existing, to be extended): Serves as authentication authority, provides presigned URLs, manages folder permissions, lists files.
- **S3 Bucket** (existing): Stores original images, `.thumbnails.json`, and generated thumbnails. Existing Lambda triggers (`upload-handler`) continue to run on new uploads.
- **Local Device DB** (in Flutter app): Tracks uploaded photos to avoid duplicates (using hash or path+size+mtime).

### Data Flow (Hourly Sync)
1. Background task wakes up.
2. App refreshes JWT if necessary.
3. Calls `/api/sync/folders` to get allowed upload prefixes for the user.
4. Uses `photo_manager` to query recent or new assets from device galleries.
5. Filters assets not present in local "uploaded" database.
6. Requests presigned PUT URL from API (`/api/sync/presign`).
7. Uploads directly to S3 using the presigned URL.
8. Records successful upload in local DB.
9. (Optional) Calls confirmation endpoint.
10. Existing AWS Lambda generates thumbnail and updates `.thumbnails.json`.

### Key Design Decisions
- **No long-lived AWS credentials on mobile** — all uploads via short-lived presigned URLs generated by the Next.js backend.
- **Authentication**: JWT (access + refresh tokens) instead of current iron-session cookies.
- **Deduplication**: Content-based hashing (SHA-256) or strong metadata key stored locally in Isar/Drift DB.
- **Scheduling**: `workmanager` (primary) + `android_alarm_manager_plus` for reliable hourly execution with battery/network constraints.

## 2. Changes to the Existing Web UI (Next.js)

Current state relies on:
- Iron-session cookies + per-folder `.password.json`
- Hardcoded S3 credentials in `.env`
- Endpoints under `app/api/s3/*` and legacy `pages/api/*`

### Required Modifications
- Introduce JWT authentication middleware (using `jose` or `next-auth`).
- Protect all S3-related endpoints with JWT validation + prefix authorization check.
- Migrate from per-folder passwords to user-based ACLs (user → list of allowed prefixes).
- Add new API routes (see below).
- Update frontend login flow to handle JWT (httpOnly cookies where possible).
- Make bucket/prefix handling more dynamic and permission-aware.
- (Optional) Add user management UI for admins.

### New API Endpoints

All new endpoints should be under `app/api/` (App Router) and require valid JWT.

- `POST /api/auth/login` — Returns `{ accessToken, refreshToken }` (update existing).
- `POST /api/auth/refresh` — Exchanges refresh token for new access token.
- `GET /api/auth/me` — Returns user profile + allowed prefixes.
- `GET /api/sync/folders` — List of prefixes the authenticated user can upload into.
- `POST /api/sync/presign` — Body: `{ prefix, filename, contentType, size? }` → Returns presigned S3 PUT URL with conditions.
- `POST /api/sync/confirm` (optional) — Notify backend of successful upload (for immediate `.thumbnails.json` update).
- Enhance existing:
  - `GET/POST /api/s3/files/[prefix]/...`
  - `POST /api/s3/upload/route.ts`
  - `DELETE /api/s3/deleteMany/route.ts`

**Presigned URL Generation** (critical):
- Use AWS SDK `getSignedUrl` with `PutObjectCommand`.
- Add conditions: content-length-range, content-type, key prefix.
- Expiration: 10–30 minutes.

## 3. Authentication Strategy

**Recommended Approach: JWT + Refresh Tokens**
- Short-lived access token (15–60 minutes) containing:
  - `userId`
  - `allowedPrefixes: string[]`
  - `exp`, `iat`
- Refresh token (longer lived, stored securely on device, rotated).
- Flutter uses `flutter_secure_storage` + `http` interceptor for automatic refresh.
- Web UI can continue using cookies but should transition to JWT validation.
- Future option: Integrate **Supabase Auth**, **Clerk**, or **Firebase** for easier mobile SDKs and social login.

**S3 Permissions**
- Next.js backend holds IAM credentials (current `.env` keys).
- Mobile never sees AWS keys.
- Bucket policy remains permissive for the server role.
- Presigned URLs are scoped per-object and time-limited.

**Security Considerations**
- Use HTTPS everywhere.
- Implement rate limiting on upload endpoints.
- Validate file types/sizes on presign.
- Store refresh tokens encrypted.
- Support multi-device usage via per-device upload records or central manifest.
- Audit logging for uploads.

## 4. Flutter Android Application

### Core Packages
- `flutter`
- `workmanager` — Background task scheduling
- `photo_manager` — Access device photos, albums, metadata
- `flutter_secure_storage` — Secure token storage
- `dio` or `http` — API client with interceptors
- `isar` or `drift` — Local database for uploaded photo manifest
- `permission_handler`
- `path_provider`, `crypto` (for hashing), `path`
- `flutter_local_notifications` — Progress / completion alerts

### App Features
- Onboarding: Login, select default upload prefix, choose watched albums.
- Settings screen: Sync interval, watched folders, last sync time, "Sync Now" button.
- Background sync task (hourly):
  1. Authenticate / refresh token.
  2. Fetch allowed prefixes.
  3. Query new photos (modified since last sync or not in DB).
  4. Compute hash or use metadata key.
  5. Request presigned URL.
  6. Upload (with progress).
  7. Store success in local DB.
- Manual trigger from UI.
- Error queue with retry.

### Implementation Priorities
1. Auth service + JWT management.
2. Local database schema (uploaded photos).
3. Photo scanning with `photo_manager`.
4. Presigned upload flow.
5. WorkManager background task.
6. UI (settings + sync status).

## 5. Implementation Roadmap

**Phase 1: Backend Foundation**
- Add JWT auth layer and middleware.
- Implement `/api/sync/presign` endpoint (highest priority).
- Protect existing S3 routes with authorization.
- Simple user-to-prefix mapping (can start in S3 or env).

**Phase 2: Mobile MVP**
- Create Flutter project.
- Implement login + token refresh.
- Build "Sync Now" feature (single folder).
- Add local DB and deduplication.
- Register background task.

**Phase 3: Polish & Production**
- Full multi-folder support.
- Progress UI and notifications.
- Web UI updates (new login, admin controls).
- Battery optimization handling, permission flows.
- Monitoring / logging.
- Multi-device conflict resolution.

**Phase 4 (Optional)**
- Migrate to managed auth provider (Supabase/Clerk).
- Add video support.
- Web UI enhancements (upload queue, device list).

## Security & Operational Notes

- **Never** embed AWS credentials in the Flutter app.
- Respect Android Doze / battery optimization — document why the app needs background access.
- Use strong deduplication to prevent duplicate uploads across devices.
- Monitor S3 costs (thumbnails + storage).
- Consider adding upload quotas per user.
- Backup local upload DB or make it recoverable.
